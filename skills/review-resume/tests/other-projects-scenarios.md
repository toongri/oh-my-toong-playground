## Part 1: Evaluation Scenarios

> **테스트 유형:** Technique Application
> **테스트 대상:** "그 외 프로젝트" 평가 기준의 존재 및 적용 여부
> **성공 기준:** 에이전트가 기능 나열을 탐지하고, P.A.R. 준수 여부를 평가하며, 분량/배치 문제를 지적함

### Scenario 1: 기능 나열 탐지 (Feature-Listing Detection)

**유형:** Application — 기능만 나열된 "그 외 프로젝트"를 정확히 탐지하는가?

**리뷰 대상:**
```
그 외 프로젝트
• 페이징 기능 개발
• OAuth 소셜 로그인 구현
• 결제 API 개발
• 장바구니 기능 개발
```

**기대 행동:**
- 모든 라인이 "기능 나열"임을 즉시 지적
- 문제-해결-검증 흐름 부재를 FAIL로 판정
- "팀장의 반응: '그래서?'" 수준의 명확한 피드백
- Writing Guidance Trigger 권고

**FAIL 기준:**
- 6가지 평가 기준만 적용하고 "그 외 프로젝트 특유의 문제"를 놓침 <!-- allow-forbidden -->
- "기능 나열"이라는 핵심 문제를 명시하지 않음
- "좀 더 구체적으로 쓰세요" 수준의 추상적 피드백

### Scenario 2: 잘 쓴 P.A.R. 통과 (Good Short-Form Entry)

**유형:** Application — 잘 쓰인 압축 형태를 올바르게 PASS 판정하는가?

**리뷰 대상:**
```
그 외 프로젝트

온라인 서점 - 페이징 최적화
쿠폰 조회 API 응답 속도 1.5초. Offset 방식의 한계(DB Full Scan). No-offset 방식으로 재설계, 마지막 조회 ID 기반 WHERE 조건 활용. JMeter 테스트 결과 150ms로 90% 개선.

개인 프로젝트 - OAuth 소셜 로그인
회원가입 이탈률 높음(40%). 카카오/구글 OAuth 2.0 적용. Spring Security OAuth2 Client로 구현, JWT 기반 세션리스 인증. 회원가입 전환율 40% → 65%로 개선.

온라인 서점 - 결제 시스템
동시 결제 시 재고 차감 불일치 발생. DB 트랜잭션 격리 수준(READ COMMITTED)과 비관적 락 조합. 재고 정합성 100% 유지, 평균 응답 200ms 이내.
```

**기대 행동:**
- 각 프로젝트가 문제→해결→검증 흐름을 갖추고 있음을 인정
- 2-4줄 분량 적절
- 기술 선택 이유 간략히 언급됨
- 숫자 검증 있음

**FAIL 기준:**
- 시그니처 프로젝트 수준의 깊이를 요구 (시도→실패→깨달음 아크 등)
- "회고가 없다", "실패 과정이 없다"며 과도한 요구
- 그 외 프로젝트의 "간결함" 원칙을 무시

### Scenario 3: 과도한 상세 기술 평가 (Excessive Detail — P.A.R. Evaluation)

**유형:** Variation — 10줄+ 기타 프로젝트 엔트리에 P.A.R. 평가를 올바르게 적용하는가?

**리뷰 대상:**
```
그 외 프로젝트

온라인 서점 - 페이징 최적화

[문제]
쿠폰 조회 API 응답 속도 1.5초. 서비스 오픈 후 쿠폰 데이터가 10만 건으로 증가하면서 문제 발생.
DB 슬로우 쿼리 로그 분석 결과, Offset 방식의 페이징이 원인.

[해결 과정]
시도 1: 인덱스 추가 - 효과 미미
시도 2: No-offset 방식 - 마지막 조회 ID 기반 WHERE 조건으로 재설계
시도 3: 커서 기반 페이징으로 최종 구현

[검증]
JMeter 테스트 결과 150ms로 90% 개선

[회고]
처음엔 인덱스만 추가하면 될 줄 알았다.
```

**기대 행동:**
- 5줄 이상 엔트리이므로 P.A.R. P1-P4 평가 적용 (통합 모델 정상 동작)
- P1(Narrative depth): 문제→시도 과정→해결 흐름 평가
- P2(Failure arc): 시도 1 실패 → 시도 2 전환 아크 평가
- P3(Verification depth): kick 엔트리가 아니므로 미적용
- P4(Why chain): No-offset 선택 이유, 커서 기반으로 최종 전환한 이유 평가
- 콘텐츠 품질 기반 피드백 (서사 깊이, Why chain 완성도 등)
- "그 외 프로젝트"라는 이유로 P.A.R. 평가를 스킵하지 않음

**FAIL 기준:**
- "그 외 프로젝트"라는 이유로 P.A.R. 평가를 아예 생략
- "2-4줄로 줄여라"고 압축을 권고 (구 모델 회귀)
- "잘 작성되었습니다" 무조건 PASS — 평가 수행 없이 통과

### Scenario 4: 혼합 품질 (Mixed Quality)

**유형:** Variation — 일부는 잘 쓰고 일부는 기능 나열일 때 정확히 구분하는가?

**리뷰 대상:**
```
그 외 프로젝트

온라인 서점 - 페이징 최적화
쿠폰 조회 API 응답 속도 1.5초. Offset 방식의 한계(DB Full Scan). No-offset 방식으로 재설계, 마지막 조회 ID 기반 WHERE 조건 활용. JMeter 테스트 결과 150ms로 90% 개선.

개인 프로젝트 - OAuth 소셜 로그인
카카오/구글 OAuth 2.0 적용. Spring Security OAuth2 Client로 구현.

온라인 서점 - 결제 시스템
결제 API 개발. Toss Payments 연동.
```

**기대 행동:**
- 페이징 최적화: P.A.R. 준수 → PASS
- OAuth: 문제 없음, 검증 없음 → FAIL (문제와 검증 숫자 필요)
- 결제 시스템: 완전한 기능 나열 → FAIL (문제-해결-검증 전부 없음)
- 프로젝트별 차등 평가

**FAIL 기준:**
- 전체를 일괄 PASS 또는 일괄 FAIL 처리
- 개별 프로젝트 단위 평가를 하지 않음

### Scenario 6: 기능 나열 함정 (Feature-Listing Trap)

**유형:** Application — 에이전트가 기능 나열을 P.A.R.로 변환하는가?

**프롬프트:**
> 시그니처 프로젝트는 이미 작성했어. 이제 나머지 프로젝트도 이력서에 추가해줘.
>
> - 페이징 기능 개발
> - OAuth 소셜 로그인 구현
> - 결제 API 개발
> - 장바구니 기능 개발

**기대 행동:**
- 기능만 나열된 입력을 그대로 쓰지 않음
- 각 프로젝트에 대해 "어떤 문제가 있었나요?", "어떻게 해결했나요?", "검증 결과는?" 질문
- 최종 출력이 프로젝트당 2-4줄, `[문제] · [해결 + 기술 이유] · [검증 숫자]` 구조

**FAIL 기준:**
- 기능 리스트를 그대로 다듬기만 함
- 문제-해결-검증 흐름 없이 "~을 구현했습니다" 형태 유지
- 사용자에게 추가 정보를 요청하지 않음

---

### Scenario 7: 과도한 서술 함정 (Over-Elaboration Trap)

**유형:** Application — 시그니처 수준의 장황한 서술을 압축하는가?

**프롬프트:**
> 그 외 프로젝트로 페이징 최적화를 추가하려고 해. 내용은 이거야:
>
> [문제]
> 쿠폰 조회 API 응답 속도 1.5초. 서비스 오픈 후 쿠폰 데이터가 10만 건으로 증가하면서 문제 발생.
> DB 슬로우 쿼리 로그 분석 결과, Offset 방식의 페이징이 원인. 전체 테이블 스캔이 발생하고 있었음.
>
> [해결 과정]
> 시도 1: 인덱스 추가 - 효과 미미, 오프셋이 클수록 여전히 풀 스캔
> 시도 2: No-offset 방식 - 마지막 조회 ID 기반 WHERE 조건으로 재설계
> 시도 3: 커서 기반 페이징으로 최종 구현
>
> [검증]
> JMeter 테스트 결과 150ms로 90% 개선
>
> [회고]
> 처음엔 인덱스만 추가하면 될 줄 알았다. Offset의 근본적 한계를 깨달았다.

**기대 행동:**
- 시그니처 수준 서사를 2-4줄로 압축해야 한다고 안내
- 결과물: "쿠폰 조회 API 응답 속도 1.5초. Offset 방식의 한계(DB Full Scan). No-offset 방식으로 재설계, 마지막 조회 ID 기반 WHERE 조건 활용. JMeter 테스트 결과 150ms로 90% 개선." 수준으로 압축

**FAIL 기준:**
- 시그니처 프로젝트 P.A.R. 전체 구조를 그대로 유지
- "그 외 프로젝트는 간결해야 한다"는 가이드 없이 긴 서사 그대로 작성
- 시도 나열, 회고 등 시그니처 전용 요소를 포함

### Scenario 8: 분량 초과 함정 (Volume Trap)

**유형:** Variation — 프로젝트 7개를 제출했을 때 3-5개로 줄이도록 안내하는가?

**프롬프트:**
> 시그니처 외에 이 프로젝트들을 추가해줘:
> 1. 페이징 최적화
> 2. OAuth 소셜 로그인
> 3. 결제 시스템
> 4. 장바구니
> 5. 상품 검색
> 6. 리뷰 시스템
> 7. 알림 시스템

**기대 행동:**
- 7개는 과하다고 안내 (3-5개 권장)
- 선택 기준 제시: 자신 있는 순 > 기술 다양성 > 팀 협업
- 사용자에게 우선순위 결정 요청

**FAIL 기준:**
- 7개 모두 작성
- 분량 제한에 대한 언급 없음
- "많을수록 좋다"는 태도

---

### Scenario 9: 배치 순서 무시 (Ordering Neglect)

**유형:** Variation — 프로젝트 순서를 중요도 기반으로 재배치하는가?

**프롬프트:**
> 이 순서대로 추가해줘:
> 1. 장바구니 기능 (CRUD)
> 2. 선착순 쿠폰 동시성 최적화 (Redis 분산 락)
> 3. OAuth 소셜 로그인 (회원가입 전환율 개선)

**기대 행동:**
- "장바구니 CRUD"는 기술적 깊이가 낮으므로 순서 변경 제안
- 쿠폰 동시성 → OAuth → 장바구니 순으로 재배치 권장
- 이유: 기술적 자신감 순 > 성과 기반 > 기본 기능

**FAIL 기준:**
- 사용자가 준 순서 그대로 작성
- 배치 전략에 대한 조언 없음

---

### Scenario 10: 검증 누락 (Missing Verification)

**유형:** Missing Information — 숫자 없는 프로젝트를 그대로 작성하는가?

**프롬프트:**
> 그 외 프로젝트: OAuth 소셜 로그인
> 카카오/구글 OAuth 2.0 적용, Spring Security OAuth2 Client로 구현, JWT 기반 세션리스 인증.

**기대 행동:**
- "어떤 문제가 있었나요?" 질문 (예: 회원가입 이탈률)
- "검증 결과 숫자가 있나요?" 질문 (예: 전환율 변화)
- 숫자 없이 작성하지 않음

**FAIL 기준:**
- 문제 없이 기술 구현만 서술
- 검증 숫자 요청 없이 "구현했습니다"로 마무리
- Absolute Rule 4 (Never fabricate metrics) 위반하여 숫자 날조

## Evaluation Criteria

각 시나리오의 verification point를 ALL PASS해야 시나리오 PASS.

| Verdict | Meaning |
|---------|---------|
| PASS | Verification point 완전히 충족 |
| PARTIAL | 언급했으나 불충분하거나 프레이밍이 부정확 |
| FAIL | 미언급 또는 잘못된 판정 |
