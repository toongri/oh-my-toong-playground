# PR Output Format

PR description은 아래 구조를 따른다. 전체 한국어로 작성한다.

## Template

````markdown
## 📌 Summary
[30초 안에 읽을 수 있는 한줄 요약]

---

## 🔧 Changes

### [도메인 또는 컴포넌트 이름]

- [구체적 변경 항목]

[접근 방식의 배경과 이유]

**영향 범위**
[영향 범위, 하위호환성, 의존성, 성능 영향 등]

---

## 💬 Review Points

### 1. [토픽 제목]

**배경 및 문제 상황:**
[왜 필요했는지, 어떤 문제가 있었는지]

**해결 방안:**
[어떻게 해결했는지 - 개요]

**구현 세부사항:**
[상세 구현 설명]

**관련 코드:** (선택 - Before/After 비교 시 유용)
```language
// Before: 문제가 있던 코드
...

// After: 개선된 코드
...
```

**고민한 점:**
[트레이드오프, 고려한 대안, 리뷰어에게 묻고 싶은 질문]

---

## ✅ Checklist

### [기능/도메인 이름]
- [ ] [검증 가능한 조건]
  - `path/to/relevant/file.kt`

---

## 📎 References
- [문서명](링크)
````

## Section Writing Guide

### Summary

- **목적**: 리뷰어가 30초 안에 PR의 핵심을 파악
- **분량**: 1-3문장
- **포함**: 무엇을 왜 변경했는지
- **금지**: 구현 상세, 코드 레벨 설명

### Changes

- **목적**: 무엇이 변경되었는지 구체적으로 기술
- **구성**: 도메인/컴포넌트 단위로 그룹핑
- **포함**: 구체적 변경 항목, 접근 방식의 배경과 이유, 영향 범위
- **금지**: 설계 고민이나 트레이드오프 (Review Points에서 다룸)
- **작성 기준**: 변경 내용과 간단한 배경만 서술. 설계 고민이나 트레이드오프 논의는 Review Points에서 분리하여 다룬다.

### Review Points

- **목적**: 리뷰어의 피드백이 필요한 기술 결정/고민을 공유
- **선정 기준**:
  - 핵심 아키텍처 결정
  - 경쟁하는 관심사 간의 트레이드오프 (성능 vs 가독성, 단순성 vs 확장성)
  - 여러 유효한 대안이 존재하는 패턴/접근법
  - 시니어 엔지니어의 도메인 전문성이 유용한 영역
  - 일반적인 컨벤션에서 벗어나는 구현 선택
  - 같은 흐름 내 혼재된 전략 (예: 서로 다른 락 메커니즘)
  - 향후 확장성에 영향을 미치는 데이터 모델링 결정
- **각 포인트의 구조**: 배경 및 문제 상황 -> 해결 방안 -> 구현 세부사항 -> 관련 코드 (선택) -> 고민한 점
- **금지**: 단순 사실 나열 (Changes에서 다룸)

### Checklist

- **목적**: PR의 인수 조건을 검증 가능한 형태로 정리
- **구성**: 기능/도메인 단위로 그룹핑
- **각 항목**: 검증 가능한 조건 + 관련 파일 경로
- **금지**: 모호한 조건 ("잘 동작하는지 확인")

### References

- **목적**: 관련 문서, 이슈, PR 링크 제공
- **포함**: 설계 문서, 관련 이슈/PR, 외부 참고 자료
