### ST-1: 도구/기술 주어 + 수동태 복합 감지

**Technique Under Test**: T1 도구/기술 주어 금지, T2 수동태→능동태 (sentence.md Step 1)

**Input**:
```markdown
이 프레임워크는 REST API 구축을 지원합니다. 요청이 수신되면 미들웨어에 의해 처리가 수행됩니다. 응답 형식은 JSON으로 자동 변환됩니다.
```

**Expected Output**:
T1 위반:
- "이 프레임워크는 REST API 구축을 지원합니다" → 프레임워크가 주어
- 개선: "이 프레임워크를 사용하면 REST API를 구축할 수 있습니다."

T2 위반:
- "요청이 수신되면 미들웨어에 의해 처리가 수행됩니다" → 수동태
- 개선: "요청이 들어오면 미들웨어가 처리합니다."
- "응답 형식은 JSON으로 자동 변환됩니다" → 수동태
- 개선: "프레임워크가 응답을 JSON으로 자동 변환합니다."

**Pass Criteria**:
(1) T1 위반 1건 이상 지적
(2) T2 위반 1건 이상 지적
(3) 각각에 대해 Before/After 개선안 제시
(4) 3문장 중 위반 지적 0건이면 RED

### ST-2: 메타 담화 제거

**Technique Under Test**: T4 메타 담화 제거 (sentence.md Step 2)

**Input**:
```markdown
앞서 살펴보았듯이, 데이터베이스 인덱싱은 쿼리 성능에 큰 영향을 미칩니다. 이제 인덱스 생성 방법에 대해 알아보겠습니다. 간단히 말하면, B-Tree 인덱스가 가장 범용적입니다. 결론적으로, 아시다시피 적절한 인덱스 전략이 중요합니다.
```

**Expected Output**:
T4 위반 4건:
1. "앞서 살펴보았듯이" → 제거
2. "이제 ~ 알아보겠습니다" → 제거, 바로 내용 시작
3. "간단히 말하면" → 제거
4. "결론적으로, 아시다시피" → 제거

개선:
> 데이터베이스 인덱싱은 쿼리 성능에 큰 영향을 미칩니다. B-Tree 인덱스가 가장 범용적입니다. 적절한 인덱스 전략이 중요합니다.

**Pass Criteria**:
(1) 메타 담화 표현 3개 이상 지적
(2) 메타 담화 제거 후 개선 문장 제시
(3) 메타 담화를 1개도 지적하지 않으면 RED

### ST-3: 명사형 → 동사형 전환

**Technique Under Test**: T5 명사 대신 동사 사용 (sentence.md Step 3)

**Input**:
```markdown
서버 구동 확인 후 데이터베이스 연결 설정 수행이 필요합니다. 로그 파일 생성 및 모니터링 설정 진행이 완료되면 서비스 배포 실행이 가능합니다.
```

**Expected Output**:
T5 위반:
- "서버 구동 확인 후 데이터베이스 연결 설정 수행이 필요합니다" → 명사형 나열
- "로그 파일 생성 및 모니터링 설정 진행이 완료되면" → 명사형 나열
- "서비스 배포 실행이 가능합니다" → 명사형

개선:
> 서버가 구동 중인지 확인한 후 데이터베이스 연결을 설정하세요. 로그 파일을 생성하고 모니터링을 설정하면 서비스를 배포할 수 있습니다.

**Pass Criteria**:
(1) 명사형 나열 패턴을 2건 이상 지적
(2) 동사형으로 전환한 Before/After 제시
(3) 명사형 문제를 지적하지 않으면 RED

### ST-4: 모호한 표현 + 주체/맥락 누락

**Technique Under Test**: T6 모호한 표현, T7 주체/맥락 생략 금지 (sentence.md Step 3)

**Input**:
```markdown
설정을 변경하면 성능이 향상될 수 있습니다. 에러가 발생하면 재시작하세요. 일부 환경에서 문제가 발생할 수 있습니다.
```

**Expected Output**:
T6 위반:
- "성능이 향상될 수 있습니다" → 어떤 성능? 얼마나?
- "일부 환경에서" → 어떤 환경?

T7 위반:
- "설정을 변경하면" → 어떤 설정? 어디서?
- "에러가 발생하면 재시작하세요" → 무엇을 재시작?
- "문제가 발생할 수 있습니다" → 어떤 문제?

개선 예시:
> `max_connections` 값을 100에서 200으로 변경하면 동시 접속 처리량이 2배 늘어납니다. 데이터베이스 연결 에러가 발생하면 애플리케이션 서버를 재시작하세요. macOS 12 이하에서 DNS 확인 지연이 발생할 수 있습니다.

**Pass Criteria**:
(1) T6 모호한 표현 1건 이상 지적
(2) T7 주체/맥락 누락 2건 이상 지적
(3) 구체화된 Before/After 제시
(4) "성능이 향상될 수 있습니다"를 문제없다고 판단하면 RED

### ST-5: 수치 부재 + 참조 맥락 누락

**Technique Under Test**: T8 구체적 수치 제시, T9 참조 맥락 명시 (sentence.md Step 3)

**Input**:
```markdown
## 캐시 설정하기

1. Redis를 설치합니다.
2. 캐시 TTL을 설정합니다.
3. 앞서 설정한 값을 사용해 캐시를 적용합니다.

캐시를 적용하면 응답 속도가 빨라집니다. 데이터가 많을수록 효과가 큽니다.
```

**Expected Output**:
T8 위반:
- "응답 속도가 빨라집니다" → 얼마나? (예: "평균 응답 시간이 200ms에서 50ms로 줄어듭니다")
- "데이터가 많을수록" → 어느 정도? (예: "10,000건 이상의 데이터를 반복 조회할 때")

T9 위반:
- "앞서 설정한 값을 사용해" → 어떤 값? 몇 단계에서 설정한 값?
- 개선: "2단계에서 설정한 TTL 값을 사용해"

**Pass Criteria**:
(1) T8 수치 부재 1건 이상 지적
(2) T9 참조 맥락 누락 지적 ("앞서 설정한 값" 모호)
(3) 구체적 수치와 명확한 참조를 포함한 개선안 제시
(4) "앞서 설정한 값"을 문제없다고 판단하면 RED

### ST-6: 용어 불일관 + 약어 미풀이

**Technique Under Test**: T12 공식 용어, T13 일관 표현, T14 약어 풀어쓰기 (sentence.md Step 4)

**Input**:
```markdown
쿠버를 사용해 컨테이너를 배포합니다. 파드를 생성한 후 디플로이먼트를 설정합니다. 그런 다음 k8s 서비스를 만들어 pod를 노출합니다. HPA를 설정하면 자동으로 파드가 늘어납니다. CRD 기반으로 오퍼레이터를 정의합니다.
```

**Expected Output**:
T12 위반:
- "쿠버" → "쿠버네티스(Kubernetes)" (공식 명칭 사용)
- "k8s" → "쿠버네티스(Kubernetes)" (문서에서는 공식 명칭)

T13 위반:
- "파드" / "pod" 혼용 → "파드(Pod)"로 통일

T14 위반:
- "HPA" → "HPA(Horizontal Pod Autoscaler)" (첫 등장 시 풀어쓰기)
- "CRD" → "CRD(Custom Resource Definition)"

**Pass Criteria**:
(1) 비공식 약어("쿠버", "k8s") 1건 이상 지적
(2) 용어 혼용("파드"/"pod") 지적
(3) 약어 미풀이("HPA", "CRD") 1건 이상 지적
(4) "쿠버"를 문제없다고 판단하면 RED

### ST-7: 한자어 + 번역체 복합 감지

**Technique Under Test**: T10 한자어 제거, T11 번역체→동사 중심 (sentence.md Step 5)

**Input**:
```markdown
환경 변수를 이용한 설정 값 주입 처리를 수행합니다. 데이터베이스 마이그레이션 작업 진행이 완료된 후, 테스트 케이스 실행을 통한 기능 검증 절차가 이루어집니다. 빌드 프로세스 진행 시 의존성 해결 작업이 자동으로 수행됩니다.
```

**Expected Output**:
T10 위반:
- "처리를 수행합니다" → "~합니다"로 충분
- "작업 진행이 완료된 후" → "~한 후"
- "작업이 자동으로 수행됩니다" → "자동으로 ~합니다"

T11 위반:
- "환경 변수를 이용한 설정 값 주입 처리" → 번역체 명사구 나열
- "테스트 케이스 실행을 통한 기능 검증 절차가 이루어집니다" → 번역체

개선:
> 환경 변수로 설정 값을 주입합니다. 데이터베이스 마이그레이션을 완료한 후 테스트를 실행해 기능을 검증합니다. 빌드할 때 의존성을 자동으로 해결합니다.

**Pass Criteria**:
(1) T10 한자어("수행", "진행", "이루어집니다") 2건 이상 지적
(2) T11 번역체 패턴 1건 이상 지적
(3) 동사 중심으로 전환한 개선안 제시
(4) 한자어/번역체를 1개도 지적하지 않으면 RED

### ST-8: 실제 동작을 이해할 수 있게 쓰기 검증

**Technique Under Test**: T16 실제 동작을 이해할 수 있게 쓰기 (sentence.md Step 3)

**Input**:
```markdown
에러가 발생하면 Exception을 던집니다. 사용자 입력 이벤트를 발화하면 핸들러가 반응합니다. 타임아웃이 지나면 스레드를 죽이고 새 스레드를 생성합니다. 주기적으로 가비지 컬렉터가 메모리를 수거합니다.
```

**Expected Output**:
T16 위반:
- "Exception을 던집니다" → 은유적 표현
- 개선: "예외(`Exception`)를 일으킵니다."
- "이벤트를 발화하면" → 은유적 표현
- 개선: "이벤트를 보내면" 또는 "이벤트가 발생하면"
- "스레드를 죽이고" → 은유적 표현
- 개선: "스레드를 종료하고"
- "메모리를 수거합니다" → 비유적 표현
- 개선: "메모리를 해제합니다."

**Pass Criteria**:
(1) T16 위반 2건 이상 지적
(2) 각 위반에 대해 Before/After 개선안 제시
(3) 관용 표현을 실제 동작으로 변환한 표현 포함
(4) "Exception을 던집니다"를 문제없다고 판단하면 RED
