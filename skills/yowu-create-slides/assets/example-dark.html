Core Concept

VM 배포 vs 컨테이너 배포

같은 장애, 완전히 다른 복구 시간

🖥 VM 배포 — 수동 관리

  • 서버마다 직접 SSH 접속해 배포
  • 롤백은 이전 바이너리를 수동 복원
  • 배포 중 서비스 다운 불가피
  • 장애 복구 30분+
→

🐳 컨테이너 배포 — 선언형

  • 이미지 태그 하나만 바꾸면 배포
  • 롤백은 이전 이미지로 즉시 전환
  • Rolling/Blue-Green 무중단 배포
  • 장애 복구 수 분 내
Deployment Patterns

세 가지 배포 전략, 상황에 맞게

각각 서로 다른 요구사항에 답한다. 하나만 알면 선택지가 없다.

🔵
Blue-Green
현재(Blue)와 신규(Green) 환경을 동시에 운영
트래픽을 한 번에 전환, 즉각 롤백 가능
한계: 리소스 2배 필요
🐦
Canary
신규 버전에 일부 트래픽(1~5%)만 먼저 전달
실사용자로 검증 후 단계적 확대
한계: 모니터링 복잡도 증가
🔄
Rolling Update
구 버전 Pod를 순차적으로 신 버전으로 교체
리소스 절약, 자동 헬스체크
한계: 롤백 시간 소요
Container

좋은 Dockerfile과 나쁜 Dockerfile

레이어 캐싱을 이해하면 빌드 시간을 80% 줄일 수 있다

✕ 레이어 캐시 미활용
FROM node:20-alpine
WORKDIR /app
COPY . .
RUN npm install
RUN npm run build
✓ 의존성 레이어 분리
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
RUN npm run build
핵심 원칙 — 변경 빈도가 낮은 레이어를 먼저 배치한다.
package.json은 소스 코드보다 훨씬 덜 바뀐다. 순서 하나로 캐시 적중률이 90%+로 오른다.
DORA Metrics

DORA — 배포 조직의 건강 지표

Google이 제시한 4가지 지표. 이 숫자가 올라가면 비즈니스 성과도 함께 올라간다.

Deployment
배포 빈도
Elite: 하루 여러 번
Lead Time
커밋→배포 시간
Elite: 1시간 미만
MTTR
장애 복구 시간
Elite: 1시간 미만
배포 빈도가 낮으면 한 번에 많은 변경이 묶여 장애 시 원인 파악이 어렵다. 자주 배포하는 팀이 더 안전하다.
SLA / SLO

99%는 약속이 아니다 — 나인을 이해하라

SLA 99%와 99.99%의 차이는 단순히 숫자가 아니다. 연간 허용 다운타임이 87.6시간 vs 52분이다.

52분
99.99% SLA 연간 다운타임
😴 "충분히 안정적이네"
87.6h
99% SLA 연간 다운타임
🔥 3일 넘게 서비스 불가
99% = 3.65일/년
99.9% = 8.7시간/년
99.99% = 52분/년
Error Budget = 1 - SLO. 이 예산이 남아 있어야 새 기능을 배포할 수 있다. 예산이 소진되면 배포를 멈추고 안정성에 집중한다.
CI/CD

코드가 프로덕션까지 가는 여정

자동화된 파이프라인이 없으면 사람이 게이트를 지킨다. 사람은 지치고, 자동화는 지치지 않는다.

Incident

실전: "배포 후 에러율 급증"

오후 3시, 신규 배포 직후 5xx 에러율이 0.01%에서 2%로 급증. 컨테이너 배포로 빠르게 대응한다.

Step 1. 알람 수신 및 확인 Grafana
배포 이벤트 마커와 에러율 급등 그래프가 일치 → 신규 버전 문제 확정
Canary 1% 단계에서 조기 발견
Step 2. 즉시 롤백 kubectl
kubectl rollout undo deployment/api-server
이전 이미지로 Rolling Update 자동 시작. 30초 내 완료.
Step 3. 원인 분석 Traces
신규 버전의 DB 커넥션 풀 설정 오류 확인: maxPoolSize: 1
설정 파일 누락으로 기본값 1이 적용됨
Step 4. 수정 후 재배포
설정 ConfigMap 수정 후 재배포. 에러율 0.01%로 복귀.
Canary 덕분에 전체 트래픽 1%만 영향받아 신속 해결
