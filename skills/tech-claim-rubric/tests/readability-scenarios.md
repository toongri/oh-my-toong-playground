# Readability Gate — Use-Case Scenarios

Phase C (R1-R5) 평가가 정확히 작동하는지 검증하는 시나리오.

---

## Scenario 1: R1 Fail — 제거 가능한 문장이 포함된 엔트리

**Input:**
```
**문제 정의**
- LLM 검수 자동화 성공 이후 서비스 성장으로 일일 접수량 2,500→5,000건 증가, 전시 SLA(접수 후 2시간) 달성률 90%→55% 하락
- 프리미엄 브랜드(매출 기여 40%)의 전시가 일반 상품에 밀려 지연되었지만, FIFO 구조에서 우선순위 제어 불가

**기술 과제**
- 촬영 완료 시 이미지등록됨 이벤트 기반으로 7개 속성을 LLM으로 추론해야 하며, 전체 완료 후에만 최종 검수→전시 진행 가능
- 속성별 처리 시간 편차가 크다(카테고리 ~30초, 소재 10~20초, 나머지 ~5초). 단일 Consumer가 순차 처리하면 건당 p50 합계 약 80초, 5,000건 기준 111시간 분량으로 SLA 2시간은 구조적으로 불가
- 1개 속성 추론 실패 시 나머지 6개까지 폐기 후 전체 재추론 → API 비용 낭비

**해결 전략**
- Consumer 내 goroutine pool로 속성 병렬 추론, 건당 80초→30초 단축
- 프리미엄/일반 토픽 분리 + Consumer 수 차등 배치로 우선순위 구현
- 부분 실패 시 성공 결과 즉시 저장, 실패 속성만 retry 토픽에 발행
- 대사 스케줄러를 10초 주기로 운용하여 미시작+부분 완료 건 감지 복구

**결과**
- 추론 처리 시간 80초→30초/건, 전시 SLA 달성률 **55% → 95%**
- 프리미엄 브랜드 평균 전시 시간 **2시간 → 40분**
- 추론 실패 시 전체 재추론 → 실패 속성만 재처리, API 호출 비용 80% 감소
```

**Expected:**
- R1 FAIL: "기술 과제" 섹션의 3번째 bullet("1개 속성 추론 실패 시 나머지 6개까지 폐기...") 내용이 해결 전략 3번째 bullet과 결과 3번째 bullet에서 이미 전달됨. 빼도 서사에 구멍 없음.
- R1 FAIL: "대사 스케줄러를 10초 주기로 운용" — 10초라는 구현 세부가 서사에 필수가 아님. "대사 스케줄러로 미처리 건 복구" 정도면 충분.
- R3 PASS: "기술 과제" 섹션은 엔지니어링 복잡도(속성별 처리 시간 편차, 순차 처리의 구조적 한계)를 확립하는 반면, 문제 정의는 비즈니스 임팩트(SLA 달성률 하락, 프리미엄 브랜드 지연)를 다룬다. 두 섹션은 서로 다른 역할을 수행하며 내용 중복이 없다 (R1에서 잡히는 3번째 bullet 중복 제외). R3 기준상 역할이 구분된 추가 섹션은 허용된다.

## Scenario 2: R3 Pass — 설계 근거가 해결 전략에 포함된 엔트리

**Input:**
```
**문제 정의**
- 피크 시간대 상품 조회 API p95 500ms 이상으로 SLO 미달, DB CPU 90%

**해결 전략**
- Redis Cache-Aside로 DB 히트 감소
  - 리스트 캐시(TTL 5분, stale 허용)와 상품 상세 캐시(상태 변경 시 즉시 eviction) 분리
  - 리스트 miss 시 ID만 조회 후 개별 캐시 확인, miss인 상품만 batch fetch
  - 쿼리 최적화는 인덱스 이미 적용 완료로 한계, Local Cache는 다중 서버에서 상품 상태 빈번 변경 시 일관성 보장 불가
- singleflight + TTL jitter로 cache stampede 방어
  - 분산 락은 멀티 인스턴스 단일 실행을 보장하나 TTL 관리와 Redis 의존이 부담
  - singleflight는 프로세스 내 단일 실행이지만 stampede 해소에 충분하고 관리 포인트 적음
  - TTL에 jitter를 적용하여 서로 다른 키가 동시 만료되지 않도록 방지

**결과**
- 상품 조회 API p95 **500ms → 150ms**, 피크 시 DB CPU **90% → 45%**
```

**Expected:**
- R3 PASS: "쿼리 최적화는 인덱스 이미 적용 완료로 한계, Local Cache는 다중 서버에서 상품 상태 빈번 변경 시 일관성 보장 불가" — 소진된 대안을 제시하여 Cache-Aside 선택을 직접 정당화하는 설계 근거다. R3 기준상 특정 결정을 동기화하는 기술적 제약은 해결 전략 영역의 design rationale로 PASS.
- R5 PASS: 15줄, 25줄 임계값 미만 자동 PASS

## Scenario 3: R4 Fail — 기술 용어 미활용으로 장황한 엔트리

**Input:**
```
**문제 정의**
- 선착순 쿠폰 프로모션에서 동시 요청 시 정합성 보장 필요

**해결 전략**
- 쿠폰을 미리 N개 만들어 놓고, 사용자가 요청하면 잠겨있지 않은 행 하나를 골라 가져가게 했다. 이미 다른 사용자가 잡은 행은 건너뛰어 대기 시간이 제로가 됨
  - 다른 방법으로는 비관적 잠금이나 분산 잠금을 쓸 수 있지만, 결국 잠금을 잡아야 하므로 경합이 생김
- 쿠폰이 다 나간 뒤에는 캐시 서버에 매진 상태를 기록해서 데이터베이스에 아예 접근하지 않도록 했다
  - 캐시 서버가 다운돼도 데이터베이스의 상태 컬럼이 정합성을 보장

**결과**
- 첫 구매 전환 **700명**, 동시 접속 200명 처리, 초과 발급 **0건**
```

**Expected:**
- R4 FAIL: "잠겨있지 않은 행 하나를 골라 가져가게" → `SKIP LOCKED` 또는 "lock-free 병렬 발급"이라는 표준 용어가 있음. "캐시 서버" → `Redis`. "비관적 잠금이나 분산 잠금" → "비관적 락·분산 락". 범용 기술 용어를 쓰면 문장이 절반으로 줄면서 같은 깊이를 전달.
- R1 FAIL: "캐시 서버가 다운돼도 데이터베이스의 상태 컬럼이 정합성을 보장" — 방어 메커니즘의 세부로, 이 수준의 안전장치 설명은 이력서에서 빼도 서사에 구멍 없음.

## Scenario 4: R2 Fail — 메트릭이 묻혀 있는 엔트리

**Input:**
```
**문제 정의**
- POS 연동 시스템에서 외부 API 응답이 간헐적으로 10초 이상 지연되어 주문 처리가 밀림

**해결 전략**
- 외부 API 호출을 SQS로 비동기 분리하여 주문 처리 파이프라인에서 POS 응답 대기를 제거. 이로 인해 피크타임 주문~배달 완료 평균이 90분에서 67분으로 단축됨
- Circuit Breaker로 slow call 기준(5초)과 실패율을 함께 적용하여 외부 장애 시 빠른 실패 처리

**결과**
- 외부 API 장애 시에도 주문 접수 지연 없이 안정적 처리
```

**Expected:**
- R2 FAIL: 핵심 메트릭 "90분→67분"이 해결 전략 텍스트 중간에 묻혀 있음. 결과 섹션에는 정량 수치가 없고 정성적 서술만 있음. 6초 스캔 시 임팩트 파악 불가.
- R4: 경계선 — SQS, Circuit Breaker는 표준 용어 사용이지만, "slow call 기준(5초)"은 구현 세부.

## Scenario 5: All PASS — 컴팩트하고 가독성 좋은 엔트리

**Input:**
```
**문제 정의**
- 피크 시간대 상품 조회 API p95 500ms 이상으로 SLO 미달, DB CPU 90%. 매 요청마다 DB 직접 조회가 원인

**해결 전략**
- Redis Cache-Aside로 DB 히트 감소. 리스트 캐시(TTL 5분)와 상품 상세 캐시(상태 변경 시 즉시 eviction) 분리
- singleflight + TTL jitter로 cache stampede 방어 (분산 락 대비 관리 포인트 최소화)

**결과**
- 상품 조회 API p95 **500ms → 150ms**, 피크 시 DB CPU **90% → 45%**
```

**Expected:**
- R1 PASS: 모든 문장이 제거 시 서사에 구멍 남김
- R2 PASS: 문제→전략→결과 6초 내 파악 가능, 결과에 정량 메트릭 배치
- R3 PASS: 문제/전략/결과 층위 깨끗이 분리
- R4 PASS: Cache-Aside, singleflight, TTL jitter, eviction — 표준 용어 적절 활용
- R5 PASS: 약 9줄, 25줄 미만 자동 PASS

## Scenario 6: R5 Fail — 볼륨 초과 엔트리 (25줄 임계값 초과)

**Input:**
```
**문제 정의**
- 위탁판매 반품은 위탁자·플랫폼·구매자 3자 관계로, 귀책에 따라 환불금·배송비·정산이 완전히 다른 흐름을 탐. 수동 귀책 판정으로 처리 시간 3일, 주 8건 이상 분쟁 발생
- 구매자가 "상품 상태 불일치"로 반품 요청 시: ① 검수 판정 오류(플랫폼 귀책) ② 배송 중 파손(택배사 귀책) ③ 구매자 변심(구매자 귀책) — 세 가지 원인이 동일 증상으로 접수
- 귀책이 결정되면 3가지 정산 축이 동시에 분기: 구매자 환불(PG) + 반품 배송비 부담 결정 + 위탁자 정산 보류·금액 조정. PG 환불과 배송비 정산은 외부 API라 부분 실패 시 불일치 상태 발생
- 반품 상품 재입고 시 등급 재판정 필요 → 등급 하락 시 가격 재책정 → 재검수는 물리적 반품 완료 후에야 가능하여 정산 파이프라인이 반품 해소까지 대기
- 위탁자가 반품을 거부할 수 있는 계약 조건이 있어, 거부 시 플랫폼이 재고를 떠안는 리스크까지 비즈니스 로직에 포함

**기술 과제**
- 검수는 사람이 수행하고 사유를 직접 입력하므로, 비정형 텍스트에서 귀책 판별 가능한 구조화된 데이터를 뽑아내야 함
- PG 환불·택배사 정산 API는 DB 트랜잭션 경계 밖이라 부분 완료 상태가 발생하며 보상 처리가 필요
- 재검수 전까지 정산 금액을 확정할 수 없어 보류 건이 쌓이면 위탁자 전체 정산 사이클에 영향
- 위탁자 반품 거부 시 동일 반품 이벤트에서 워크플로우가 완전히 분기

**해결 전략**
- 구조화된 검수 체크리스트 + 규칙 기반 귀책 분류. 검수자가 자유 텍스트 대신 구조화된 항목으로 기록. 판매 시점 검수 기록과 비교해 시스템이 자동 분류. LLM 사진 분석도 검토했지만 물리적 상태 판정의 정확도가 금전 결정을 뒷받침하기에 부족해 기각. 명확 케이스 ~80% 자동 분류, 나머지 운영자 에스컬레이션
- 오케스트레이터 기반 보상 워크플로우. 외부 API가 DB 트랜잭션에 참여할 수 없으므로 귀책 유형별 스텝을 순차 실행하고 실패 시 역순 보상. Choreography도 검토했지만 귀책 유형에 따라 모든 후속 스텝이 달라지므로 기각. 내부 작업은 단일 DB 트랜잭션, 외부 API 실패 시에만 보상 트랜잭션 발동
- 정산 보류 + 재검수 확정 2단계. 반품 접수 시 해당 건을 정산 대상에서 보류 마킹. 잠정 환불 후 차액 정산 방식도 검토했지만 CS 리스크가 높아 기각
- 위탁자 거부 분기: 플랫폼 흡수 워크플로우. 위탁 상품 → 자사 재고 소유권 전환 → 재검수 → 등급 재판정 → 자사 채널 등록

**결과**
- 반품 처리 시간 **3일 → 당일**, 명확 귀책 자동 분류율 ~80%
- 귀책 분쟁 건수 **주 8건 → 1건**
- 외부 API 부분 실패 시 보상 트랜잭션으로 불일치 상태 해소
- 위탁자 거부 → 플랫폼 흡수 자동화로 운영자 수동 전환 작업 제거
```

**Expected:**
- R5 FAIL: ~33줄 > 25줄 임계값. Sub-bullet cover test에서 제거 가능 내용 발견 (R1 findings 참조)
- R3 FAIL: "기술 과제" bullet 2("PG 환불·택배사 정산 API는 DB 트랜잭션 경계 밖이라 부분 완료 상태가 발생하며 보상 처리가 필요")가 문제 정의 bullet 3("PG 환불과 배송비 정산은 외부 API라 부분 실패 시 불일치 상태 발생")과 cross-layer content duplication (violation type 1)
- R1 FAIL: 문제 정의 5줄 + 기술 과제 4줄 = 9줄이 해결 시작 전에 소비됨. 복수의 문장이 병합 또는 제거 가능

## Scenario 7: All Pass — 가이드라인 소폭 초과 (25줄 미만)

**Input:**
```
**문제 정의**
- 실시간 주문 이벤트를 Kafka로 수신하여 PG·재고·CRM 세 외부 시스템에 동기화하는 파이프라인에서 일 평균 이벤트 150만 건 중 피크 시 처리 지연 p99 8초, 재고 불일치 주 평균 340건 발생
- 외부 시스템 응답 지연(PG 평균 1.2초, CRM 3초)이 단일 Consumer를 블로킹하고, 외부 API 부분 실패 시 보상 경로가 없어 불일치 상태로 잔류
- 세 시스템의 처리량 한계가 달라(PG 고속·CRM 저속) 단일 처리 흐름에서 가장 느린 시스템이 전체 파이프라인 처리 속도를 결정

**해결 전략**
- 시스템별 독립 토픽(order-pg, order-inventory, order-crm) 분리 + Consumer 그룹 격리
  - PG·재고·CRM 응답 지연 차이가 크므로 단일 토픽에서는 느린 시스템이 빠른 시스템 처리를 블로킹
  - 단일 토픽 멀티 컨슈머는 파티션 수 조정만으로 시스템 간 지연 격리 불가해 기각
  - 토픽 분리 후 시스템별 Consumer 수를 독립적으로 조정 가능해 운영 유연성 확보
```
