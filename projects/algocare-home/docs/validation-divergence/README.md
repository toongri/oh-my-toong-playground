# 검증 분기 패턴 — admin 로직과 실서비스 로직이 "본질은 같고 검증만 다를" 때

같은 도메인 연산(주문 취소, 환불 등)을 admin과 실사용자가 모두 호출하는데, 실행되는 본질 로직은 같고 **검증(가능 여부 판정)만 다른** 상황을 어떻게 코드로 관리할지 정리한 문서다. 예: 실사용자는 "결제완료(CONFIRMED) 상태에서만" 취소할 수 있지만, admin은 배송 이후 상태에서도 환불을 실행할 수 있다.

## 먼저: "검증"은 한 덩어리가 아니다

이 문제를 `if (isAdmin) skipValidation()` 으로 풀면 반드시 사고가 난다. 성격이 다른 세 가지가 "검증"이라는 한 단어에 뭉쳐 있기 때문이다. 이 세 층을 갈라야 아래 패턴들이 의미를 가진다.

| 층 | 예시(환불) | 누가 깰 수 있나 | 코드 위치 |
|---|---|---|---|
| **불변식(invariant)** | 환불액 ≤ 미환불 잔액 · 이미 취소된 건 재취소 불가 · 남의 주문 못 건드림 · 이력은 반드시 남는다 | **아무도** — admin도 못 깬다 | 코어 **안쪽**, 무조건 실행 |
| **정책(policy)** | 7일 이내 · 결제완료 상태부터 · 특수 이벤트 기간 제외 · 사유 assign 완료 | 진입점마다 다름 | 코어 **바깥**, 진입점이 조립 |
| **인가(authorization)** | 이 연산을 호출할 자격이 있나 | 역할/권한이 결정 | 라우터·미들웨어(앞단) |

"admin은 프리패스"라는 말은 정확히는 **"admin은 정책층이 거의 비어 있다"**는 뜻이지 "검증이 없다"가 아니다. 실제로 PSP(Stripe·Adyen·토스)도 대시보드에서 사람이 눌러도 "환불액 ≤ 잔액", "이미 취소된 건 재취소 불가" 같은 불변식은 절대 안 뚫린다 — 대시보드가 열어주는 문은 전부 *정책* 면제이지 *불변식* 면제가 아니다.

**아래 4개 패턴은 전부 "정책층을 어디에 두느냐"의 서로 다른 답이다.** 인가(앞단)와 불변식(코어 안)은 어느 패턴을 쓰든 그 자리에 그대로 있다.

## 네 패턴과 선택 기준

이 문서들은 하나의 축의 눈금이다 — **정책을 코어에서 얼마나 멀리 밀어내나.** 왼쪽일수록 싸고, 오른쪽일수록 유연하지만 비싸다.

| 패턴 | 문서 | 언제 |
|---|---|---|
| **1. 진입점별 public 메서드 분리** | [01-method-split.md](./01-method-split.md) | 케이스 2~3개, 승인 업무 없음. **기본값** |
| **2. 정책 객체 주입 + resolver** | [02-policy-injection.md](./02-policy-injection.md) | 케이스가 3개 이상으로 번질 게 명확할 때 |
| **3. CanExecute / Execute** | [03-can-execute-execute.md](./03-can-execute-execute.md) | 프런트가 "가능 여부"를 미리 알아야 할 때(위 1·2에 얹는 옵션) |
| **4. Decider (decide/evolve)** | [04-decider.md](./04-decider.md) | 외부 액션이 도메인 로직의 **결과로서** 발생할 때. **권고사항이며 admin 검증 분기엔 부적합** |

### 선택 흐름

```
케이스가 몇 개인가?
├─ 2~3개, 앞으로도 안 늘 것 같다              → 패턴 1 (메서드 분리)
├─ 3개 이상으로 번질 게 명확하다              → 패턴 2 (정책 주입 + resolver)
└─ 사람이 승인 큐를 보고 결정하는 업무다      → 연산 자체를 분리(요청 엔티티) — 이 문서 범위 밖

그 위에 얹을지 말지(직교):
├─ 프런트가 "이 버튼 켤까"를 미리 알아야 한다 → 패턴 3 (CanExecute/Execute)을 1·2 위에 얹는다
└─ 판정을 순수 함수로 뽑아 mock 없이 테스트하고 싶다
   ├─ 외부 액션이 도메인 로직의 '결과로서' 발생 → 패턴 4 (Decider)가 잘 맞는다
   └─ 외부 액션이 도메인 로직 '안'에 존재       → 패턴 4 쓰지 말 것(순수성 포기 강제)
```

### 핵심 판정 규칙 (교차-패턴)

- **인가 판단·금액 계산은 무조건 한 벌.** admin과 user로 갈라 두 벌 유지하지 않는다. OWASP도 인가 로직 중앙화를 명시 권고한다(*"Centralize the logic for handling failed access control checks."*).
- **`isAdmin` 불리언을 코어 파라미터로 넘기지 않는다.** 이건 어느 패턴에도 속하지 않는 반패턴이다. 넘기는 건 신원(`isAdmin`)이 아니라 **연산의 성질**(정책 스위치·정책 객체)이어야 한다. 토스페이먼츠가 결제 코어에 `if (isJinyoungMall(clientKey))`를 넣었다가 배포 후 그 가맹점 결제 성공 41건이 0건이 된 실제 사고가 있다.
- **패턴 1의 메서드 이름은 상위 패턴으로 그대로 옮겨간다.** `cancelByCustomer` / `cancelByAdmin`이라는 이름은 나중에 정책 주입(2)이나 도메인 모델로 갈 때도 유지된다 — 패턴 1로 시작하는 것이 위쪽 패턴으로 가는 길을 막지 않는다.

## 진짜 유지보수 비용은 패턴 선택이 아니라 드리프트

어떤 패턴을 고르든, 실제 사고는 "두 표면(admin/user, 혹은 Node/Python)이 서서히 어긋나는" 데서 난다. 패턴 선택보다 아래 4층 방어가 유지보수성에 더 크게 기여한다. 위쪽일수록 싸고 확실하다.

1. **컴파일러에게 시키기** — 코어에 새 상태를 추가하면 모든 진입점이 컴파일 에러를 내도록 `const _exhaustive: never = x` 패턴. 상태 집합·임계값은 상수로 뽑아 양쪽이 같은 것을 import(리터럴 복제가 가장 흔한 사고 원인).
2. **표를 공유하기** — 정책을 결정표로 만들고 진입점이 그 표를 참조하면 표 자체가 SSOT가 된다.
3. **차등 테스트** — "표면 A의 판정 vs 표면 B의 판정"을 같은 입력에 대해 비교하는 테스트 하나.
4. **정책 모듈에만 뮤테이션 테스트** — 가드 한 줄을 지웠는데 테스트가 통과하면 그 가드는 미검증이다.
