# Make Resume — Professional Signature Project Test Scenarios

## Purpose

이 시나리오들은 make-resume 스킬이 **현업 시그니처 프로젝트**를 올바르게 안내하는지 검증합니다.
핵심 차이: 부트캠프 시그니처(CS 깊이)와 현업 시그니처(엔지니어링 판단력)를 구분하는가?

## Technique Coverage Map

| # | Scenario | Primary Target | Secondary |
|---|---------|---------------|-----------|
| 1 | 현업 3년차 + 정답 없는 문제 | 부트캠프 vs 현업 프레이밍 구분 | 엔지니어링 판단력 강조 |
| 2 | 현업 시도의 Why — 도메인 실패 이유 | 각 시도의 "왜 이 도메인에서 안 되는지" | CS 원리 vs 도메인 이유 구분 |
| 3 | 멈추는 판단 | "할 수 있지만 안 한다"의 가치 인식 | 비용 대비 판단 |
| 4 | 비즈니스 임팩트 압도적 강조 | 수치 기반 비즈니스 성과 | 실험 기반 의사결정 |

## Scenario 1: 현업 3년차 + 정답 없는 문제

**검증 대상:** 3년차 현업 개발자의 시그니처 프로젝트를 부트캠프 프레이밍(CS 깊이)이 아닌 현업 프레이밍(엔지니어링 판단력)으로 안내하는가?

**Prompt:**
```
나 3년차 백엔드 개발자야. 시그니처 프로젝트 써줘.

메뉴 사진에서 음식 메타데이터를 자동 추출하는 LLM 기반 시스템을 만들었어.
5개 모델 비교해서 최적 조합 찾았고, 정확도 85% 달성.
수작업 11명→3명으로 줄이고, 재고 파악 4주→1주로 단축했어.
```

**Verification Points:**

| # | Check | Expected Behavior |
|---|-------|-------------------|
| V1 | 현업 프레이밍 적용 | "엔지니어링 판단력을 보여주겠다" 관점. "CS 깊이를 보여주겠다"가 아님 |
| V2 | 정답 없는 문제 인식 | 동시성(정답 있음: 락) vs LLM 정확도(정답 없음) 차이 인식. 이 문제의 본질은 "정답이 없는 영역에서 답을 찾아가는 과정" |
| V3 | 실험 기반 의사결정 유도 | "5개 모델 비교"를 수치 기반 판단/검증 서사로 확장 요구 |
| V4 | 비즈니스 임팩트 강조 | 인력 11→3명, 재고 4주→1주를 핵심 성과로 부각 |
| V5 | P.A.R.R. 구조 유지 | 현업 프레이밍이라도 시도→실패→깨달음 아크는 필수 |

## Scenario 2: 현업 시도의 Why — 도메인 실패 이유

**검증 대상:** 각 시도의 실패 이유를 "CS 원리"가 아닌 "왜 이 접근이 이 도메인에서 안 되는지"로 설명하도록 안내하는가?

**Prompt:**
```
3년차 현업이야. 시그니처 프로젝트 해결 과정:

시도 1: 규칙 기반 파싱 → 실패
시도 2: 단일 LLM → 실패
시도 3: 2단계 파이프라인 → 성공
```

**Verification Points:**

| # | Check | Expected Behavior |
|---|-------|-------------------|
| V1 | 각 시도에 "왜 이 도메인에서 안 되는가?" | 시도 1: "왜 규칙 기반이 메뉴 도메인에서 안 되는가?" 시도 2: "왜 단일 LLM이 부족한가?" |
| V2 | CS 원리 아닌 도메인 이유 | "MVCC 특성상 필연적"이 아닌 "메뉴 사진의 다양성/비정형성 때문에 규칙으로 커버 불가" 같은 도메인 맥락 |
| V3 | 수치 기반 실패 근거 | 각 시도의 실패를 "정확도 40%", "할루시네이션 30%" 같은 수치로 요구 |
| V4 | Why 체인 유지 | "왜 규칙이 안 되나? → 왜 단일 LLM이 부족한가? → 왜 2단계인가?" Why 구조는 유지 |
| V5 | 시도마다 why 포함 요구 | 각 해결과정의 선택 이유(왜 이걸 시도했나?)와 실패 이유(왜 안 됐나?)를 모두 요구 |

## Scenario 3: 멈추는 판단

**검증 대상:** "할 수 있지만 비용 대비 안 한다"는 판단을 현업 엔지니어링의 핵심 가치로 인식하는가?

**Prompt:**
```
3년차 현업. 시그니처 프로젝트 회고 부분:

93%까지 올릴 수 있었지만 fine-tuning 비용이 월 200만원 추가. 85%에서 멈추고 수작업 보조로 대체했어.
이력서에 이거 어떻게 써?
```

**Verification Points:**

| # | Check | Expected Behavior |
|---|-------|-------------------|
| V1 | 멈추는 판단 가치 인식 | "이게 현업 엔지니어링"임을 인식. 93% 가능했지만 비용 대비 보류한 것이 강점 |
| V2 | 부트캠프와의 차이점 | 부트캠프는 "더 높은 정확도 = 더 좋은 결과". 현업은 "비용 대비 최적점을 찾는 것" |
| V3 | 회고에 자연스럽게 통합 | [회고]에 트레이드오프로 녹이도록 안내. "인정하는 한계"가 아닌 "의도적 선택" |
| V4 | 면접 스토리 연결 | "왜 93%까지 안 올렸나?"라는 면접 질문에 대한 답변 시나리오 |

## Scenario 4: 비즈니스 임팩트 + 실험 기반 의사결정

**검증 대상:** 현업 시그니처에서 비즈니스 임팩트와 실험 기반 의사결정을 핵심 서사로 안내하는가?

**Prompt:**
```
3년차 백엔드. 시그니처 프로젝트 써줘.

GPT-4, Claude, Gemini 등 5개 모델 비교해서 최적 조합 찾았어. 결과적으로 인력 11명에서 3명으로, 재고 파악 4주에서 1주로 줄었어.
```

**Verification Points:**

| # | Check | Expected Behavior |
|---|-------|-------------------|
| V1 | 실험 설계 서사화 | "5개 모델 비교"를 체계적 실험 서사로 — 평가 기준, 비교 방법, 판단 근거 |
| V2 | 비즈니스 임팩트 부각 | 인력 절감, 재고 단축을 [결과]의 핵심으로. "월 1,500만원 절감" 같은 금액 환산 유도 |
| V3 | 수치 기반 판단 강조 | 각 모델의 정확도, 비용, 속도를 수치로 비교한 과정 요구 |
| V4 | 현업 프레이밍 | "어떤 CS 지식을 적용했는가?"가 아닌 "어떤 엔지니어링 판단을 내렸는가?" |
| V5 | Why 포함 | 각 모델 선택/탈락의 이유(왜 이 모델은 안 되는지, 왜 최종 조합인지) |

---

## Evaluation Criteria

각 시나리오의 verification point를 ALL PASS해야 시나리오 PASS.

| Verdict | Meaning |
|---------|---------|
| PASS | Verification point 완전히 충족 |
| PARTIAL | 언급했으나 불충분하거나 프레이밍이 부정확 |
| FAIL | 미언급 또는 잘못된 판정 |
