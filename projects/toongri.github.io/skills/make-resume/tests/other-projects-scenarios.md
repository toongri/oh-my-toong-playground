# "그 외 프로젝트" Application Scenarios — make-resume

> **테스트 유형:** Technique Application
> **테스트 대상:** "그 외 프로젝트" 작성 가이드의 존재 및 적용 여부
> **성공 기준:** 에이전트가 기능 나열을 거부하고, 압축 P.A.R.R.로 유도하며, 분량/배치를 제어함

---

## Scenario 1: 기능 나열 함정 (Feature-Listing Trap)

**유형:** Application — 에이전트가 기능 나열을 압축 P.A.R.R.로 변환하는가?

**프롬프트:**
> 시그니처 프로젝트는 이미 작성했어. 이제 나머지 프로젝트도 이력서에 추가해줘.
>
> - 페이징 기능 개발
> - OAuth 소셜 로그인 구현
> - 결제 API 개발
> - 장바구니 기능 개발

**기대 행동:**
- 기능만 나열된 입력을 그대로 쓰지 않음
- 각 프로젝트에 대해 "어떤 문제가 있었나요?", "어떻게 해결했나요?", "검증 결과는?" 질문
- 최종 출력이 프로젝트당 2-4줄, `[문제] · [해결 + 기술 이유] · [검증 숫자]` 구조

**FAIL 기준:**
- 기능 리스트를 그대로 다듬기만 함
- 문제-해결-검증 흐름 없이 "~을 구현했습니다" 형태 유지
- 사용자에게 추가 정보를 요청하지 않음

---

## Scenario 2: 과도한 서술 함정 (Over-Elaboration Trap)

**유형:** Application — 시그니처 수준의 장황한 서술을 압축하는가?

**프롬프트:**
> 그 외 프로젝트로 페이징 최적화를 추가하려고 해. 내용은 이거야:
>
> [문제]
> 쿠폰 조회 API 응답 속도 1.5초. 서비스 오픈 후 쿠폰 데이터가 10만 건으로 증가하면서 문제 발생.
> DB 슬로우 쿼리 로그 분석 결과, Offset 방식의 페이징이 원인. 전체 테이블 스캔이 발생하고 있었음.
>
> [해결 과정]
> 시도 1: 인덱스 추가 - 효과 미미, 오프셋이 클수록 여전히 풀 스캔
> 시도 2: No-offset 방식 - 마지막 조회 ID 기반 WHERE 조건으로 재설계
> 시도 3: 커서 기반 페이징으로 최종 구현
>
> [검증]
> JMeter 테스트 결과 150ms로 90% 개선
>
> [회고]
> 처음엔 인덱스만 추가하면 될 줄 알았다. Offset의 근본적 한계를 깨달았다.

**기대 행동:**
- 시그니처 수준 서사를 2-4줄로 압축해야 한다고 안내
- 결과물: "쿠폰 조회 API 응답 속도 1.5초. Offset 방식의 한계(DB Full Scan). No-offset 방식으로 재설계, 마지막 조회 ID 기반 WHERE 조건 활용. JMeter 테스트 결과 150ms로 90% 개선." 수준으로 압축

**FAIL 기준:**
- 시그니처 프로젝트 P.A.R.R. 전체 구조를 그대로 유지
- "그 외 프로젝트는 간결해야 한다"는 가이드 없이 긴 서사 그대로 작성
- 시도 나열, 회고 등 시그니처 전용 요소를 포함

## Scenario 3: 분량 초과 함정 (Volume Trap)

**유형:** Variation — 프로젝트 7개를 제출했을 때 3-5개로 줄이도록 안내하는가?

**프롬프트:**
> 시그니처 외에 이 프로젝트들을 추가해줘:
> 1. 페이징 최적화
> 2. OAuth 소셜 로그인
> 3. 결제 시스템
> 4. 장바구니
> 5. 상품 검색
> 6. 리뷰 시스템
> 7. 알림 시스템

**기대 행동:**
- 7개는 과하다고 안내 (3-5개 권장)
- 선택 기준 제시: 자신 있는 순 > 기술 다양성 > 팀 협업
- 사용자에게 우선순위 결정 요청

**FAIL 기준:**
- 7개 모두 작성
- 분량 제한에 대한 언급 없음
- "많을수록 좋다"는 태도

---

## Scenario 4: 배치 순서 무시 (Ordering Neglect)

**유형:** Variation — 프로젝트 순서를 중요도 기반으로 재배치하는가?

**프롬프트:**
> 이 순서대로 추가해줘:
> 1. 장바구니 기능 (CRUD)
> 2. 선착순 쿠폰 동시성 최적화 (Redis 분산 락)
> 3. OAuth 소셜 로그인 (회원가입 전환율 개선)

**기대 행동:**
- "장바구니 CRUD"는 기술적 깊이가 낮으므로 순서 변경 제안
- 쿠폰 동시성 → OAuth → 장바구니 순으로 재배치 권장
- 이유: 기술적 자신감 순 > 성과 기반 > 기본 기능

**FAIL 기준:**
- 사용자가 준 순서 그대로 작성
- 배치 전략에 대한 조언 없음

---

## Scenario 5: 검증 누락 (Missing Verification)

**유형:** Missing Information — 숫자 없는 프로젝트를 그대로 작성하는가?

**프롬프트:**
> 그 외 프로젝트: OAuth 소셜 로그인
> 카카오/구글 OAuth 2.0 적용, Spring Security OAuth2 Client로 구현, JWT 기반 세션리스 인증.

**기대 행동:**
- "어떤 문제가 있었나요?" 질문 (예: 회원가입 이탈률)
- "검증 결과 숫자가 있나요?" 질문 (예: 전환율 변화)
- 숫자 없이 작성하지 않음

**FAIL 기준:**
- 문제 없이 기술 구현만 서술
- 검증 숫자 요청 없이 "구현했습니다"로 마무리
- make-resume의 Absolute Rule 1 (Never fabricate metrics) 위반하여 숫자 날조
