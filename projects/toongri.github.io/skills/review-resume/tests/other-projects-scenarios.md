# "그 외 프로젝트" Application Scenarios — review-resume

> **테스트 유형:** Technique Application
> **테스트 대상:** "그 외 프로젝트" 평가 기준의 존재 및 적용 여부
> **성공 기준:** 에이전트가 기능 나열을 탐지하고, 압축 P.A.R.R. 준수 여부를 평가하며, 분량/배치 문제를 지적함

---

## Scenario 1: 기능 나열 탐지 (Feature-Listing Detection)

**유형:** Application — 기능만 나열된 "그 외 프로젝트"를 정확히 탐지하는가?

**리뷰 대상:**
```
그 외 프로젝트
• 페이징 기능 개발
• OAuth 소셜 로그인 구현
• 결제 API 개발
• 장바구니 기능 개발
```

**기대 행동:**
- 모든 라인이 "기능 나열"임을 즉시 지적
- 문제-해결-검증 흐름 부재를 FAIL로 판정
- "팀장의 반응: '그래서?'" 수준의 명확한 피드백
- make-resume로 핸드오프 권고

**FAIL 기준:**
- D1-D6만 적용하고 "그 외 프로젝트 특유의 문제"를 놓침
- "기능 나열"이라는 핵심 문제를 명시하지 않음
- "좀 더 구체적으로 쓰세요" 수준의 추상적 피드백

---

## Scenario 2: 잘 쓴 압축 P.A.R.R. 통과 (Good Compressed P.A.R.R.)

**유형:** Application — 잘 쓰인 압축 형태를 올바르게 PASS 판정하는가?

**리뷰 대상:**
```
그 외 프로젝트

온라인 서점 - 페이징 최적화
쿠폰 조회 API 응답 속도 1.5초. Offset 방식의 한계(DB Full Scan). No-offset 방식으로 재설계, 마지막 조회 ID 기반 WHERE 조건 활용. JMeter 테스트 결과 150ms로 90% 개선.

개인 프로젝트 - OAuth 소셜 로그인
회원가입 이탈률 높음(40%). 카카오/구글 OAuth 2.0 적용. Spring Security OAuth2 Client로 구현, JWT 기반 세션리스 인증. 회원가입 전환율 40% → 65%로 개선.

온라인 서점 - 결제 시스템
동시 결제 시 재고 차감 불일치 발생. DB 트랜잭션 격리 수준(READ COMMITTED)과 비관적 락 조합. 재고 정합성 100% 유지, 평균 응답 200ms 이내.
```

**기대 행동:**
- 각 프로젝트가 문제→해결→검증 흐름을 갖추고 있음을 인정
- 2-4줄 분량 적절
- 기술 선택 이유 간략히 언급됨
- 숫자 검증 있음

**FAIL 기준:**
- 시그니처 프로젝트 수준의 깊이를 요구 (시도→실패→깨달음 아크 등)
- "회고가 없다", "실패 과정이 없다"며 과도한 요구
- 그 외 프로젝트의 "간결함" 원칙을 무시

## Scenario 3: 과다 분량 탐지 (Over-Verbose Detection)

**유형:** Variation — 그 외 프로젝트가 시그니처 수준으로 장황할 때 탐지하는가?

**리뷰 대상:**
```
그 외 프로젝트

온라인 서점 - 페이징 최적화

[문제]
쿠폰 조회 API 응답 속도 1.5초. 서비스 오픈 후 쿠폰 데이터가 10만 건으로 증가하면서 문제 발생.
DB 슬로우 쿼리 로그 분석 결과, Offset 방식의 페이징이 원인.

[해결 과정]
시도 1: 인덱스 추가 - 효과 미미
시도 2: No-offset 방식 - 마지막 조회 ID 기반 WHERE 조건으로 재설계
시도 3: 커서 기반 페이징으로 최종 구현

[검증]
JMeter 테스트 결과 150ms로 90% 개선

[회고]
처음엔 인덱스만 추가하면 될 줄 알았다.
```

**기대 행동:**
- "그 외 프로젝트"치고 너무 길다고 지적 (2-4줄 권장, 현재 10줄+)
- 시그니처 수준 구조([문제], [해결 과정], [검증], [회고])를 그 외 프로젝트에 쓸 필요 없다고 안내
- 압축 형태로 변환 권고

**FAIL 기준:**
- 분량 문제를 지적하지 않음
- "잘 작성되었습니다" 판정
- 시그니처 프로젝트와 동일한 P1-P5 기준으로 평가

## Scenario 4: 혼합 품질 (Mixed Quality)

**유형:** Variation — 일부는 잘 쓰고 일부는 기능 나열일 때 정확히 구분하는가?

**리뷰 대상:**
```
그 외 프로젝트

온라인 서점 - 페이징 최적화
쿠폰 조회 API 응답 속도 1.5초. Offset 방식의 한계(DB Full Scan). No-offset 방식으로 재설계, 마지막 조회 ID 기반 WHERE 조건 활용. JMeter 테스트 결과 150ms로 90% 개선.

개인 프로젝트 - OAuth 소셜 로그인
카카오/구글 OAuth 2.0 적용. Spring Security OAuth2 Client로 구현.

온라인 서점 - 결제 시스템
결제 API 개발. Toss Payments 연동.
```

**기대 행동:**
- 페이징 최적화: 압축 P.A.R.R. 준수 → PASS
- OAuth: 문제 없음, 검증 없음 → FAIL (문제와 검증 숫자 필요)
- 결제 시스템: 완전한 기능 나열 → FAIL (문제-해결-검증 전부 없음)
- 프로젝트별 차등 평가

**FAIL 기준:**
- 전체를 일괄 PASS 또는 일괄 FAIL 처리
- 개별 프로젝트 단위 평가를 하지 않음

## Scenario 5: 프로젝트 과다 탐지 (Excessive Projects)

**유형:** Variation — 6개 이상 프로젝트가 나열됐을 때 산만함을 지적하는가?

**리뷰 대상:**
```
그 외 프로젝트

온라인 서점 - 페이징 최적화
(적절한 2줄 내용)

개인 프로젝트 - OAuth 소셜 로그인
(적절한 2줄 내용)

온라인 서점 - 결제 시스템
(적절한 2줄 내용)

온라인 서점 - 장바구니
(적절한 2줄 내용)

온라인 서점 - 상품 검색
(적절한 2줄 내용)

개인 프로젝트 - 알림 시스템
(적절한 2줄 내용)

개인 프로젝트 - 리뷰 시스템
(적절한 2줄 내용)
```

**기대 행동:**
- 7개는 산만하다고 지적 (3-5개 권장)
- 우선순위 기반 선별 권고
- "존재해야 하고, 깔끔해야 하고, 과하지 않아야 한다"

**FAIL 기준:**
- 프로젝트 수에 대한 지적 없음
- 7개 전부 개별 평가만 하고 전체 분량 문제 무시
